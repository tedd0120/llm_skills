# 小红书内容抓取 Skill

一站式小红书内容抓取与分析入口。作为编排层，协调各个子 skills 完成从数据抓取到报告生成的完整流程。

**内部架构**：
- `scripts/orchestrate_login.py` - 登录编排入口，负责轮询 `login_xhs.py` 输出并收敛为稳定事件
- `scripts/login_xhs.py` - 单次调用内检查登录状态，并在需要时处理扫码登录
- `xiaohongshu-fetch` - 执行浏览器自动化抓取，输出 raw.json（详见其 SKILL.md）
- `xiaohongshu-summarize` - 分析数据并生成结构化报告（描述性 skill）
- `xiaohongshu-formatter` - 美化报告格式，增强 emoji（描述性 skill）

---

## 两阶段执行流程

当用户调用此命令时，请按以下两个阶段执行：

### 阶段一：澄清阶段（必须执行）

此阶段的目标是与用户确认搜索关键词，确保爬虫抓取的内容符合用户意图。**确认前禁止进入执行阶段**。

#### 1. 解析用户输入

- **将用户输入视为整体文本**，忽略内部所有分隔符（逗号、空格、分号等）
- 理解用户建议的主旨，识别核心要素：主题、地域、场景、意图
- **示例**：输入 "广州装修公司, 对比, 避坑" 应理解为 "广州装修公司相关" 这一整体建议

#### 2. 选择搜索模式

在解析用户输入后、衍生关键词前，**必须**先展示模式选择：

```markdown
请先选择搜索模式：

| 选项 | 模式 | 说明 |
|:----:|:-----|:-----|
| A | 固定关键词模式 | 你先确认关键词，系统按关键词一次性执行搜索 |
| B | 发散模式 | AI 从主题出发自动多轮搜索，每轮动态决定下一关键词 |

回复 `A` 或 `B`。
```

**[核心要求] 逐轮交互约束**：
- 澄清阶段每次输出只能包含一个问题
- 必须等待用户回复后再输出下一个问题
- **禁止**将多个问题合并到一次输出中

#### 3. 模式 A（固定关键词）: 逐轮交互序列

模式 A 必须按以下顺序逐轮交互，每轮只能输出一个问题：

**第 1 轮：模式选择确认**
- 用户回复 `A` 后，进入第 2 轮

**第 2 轮：篇数上限询问**
```markdown
已选择固定关键词模式。
篇数上限是多少？（默认 30，最大 100）
```

**第 3 轮：衍生结果展示 + 确认询问**

根据用户在第 2 轮回复的篇数上限，衍生搜索关键词并展示：

```markdown
我基于你的建议衍生了以下 3 个搜索词：

| 序号 | 关键词 | 预计篇数 | 搜索理由 |
|:----:|:------|:--------:|:---------|
| 1 | [关键词一] | 10篇 | [理由一] |
| 2 | [关键词二] | 10篇 | [理由二] |
| 3 | [关键词三] | 10篇 | [理由三] |
| **总计** | — | **30篇** | — |

你可以：修改单个关键词、调整数量、重新生成、或确认开始爬取。
```

等待用户回复，循环调整直到用户确认（"确认"、"开始"、"好的"、"就这样"等）。

**第 4 轮：超链接格式选择**

用户确认关键词后，询问超链接格式：

```markdown
报告中超链接格式：

| 选项 | 格式   | 说明                              |
|:----:|:------|:----------------------------------|
| A    | 纯文本  | 原帖引用为纯文本，无超链接（默认）     |
| B    | 超链接  | 原帖引用可直接点击跳转到小红书       |

回复 `A` 或 `B`（默认 A）。
```

**模式 A 交互序列总结**：
1. 模式选择确认
2. 篇数上限询问
3. 衍生结果展示 + 确认询问
4. 超链接格式选择

#### 4. 模式 B（发散模式）: 逐轮交互序列

模式 B 必须按以下顺序逐轮交互，每轮只能输出一个问题：

**第 1 轮：模式选择确认**
- 用户回复 `B` 后，进入第 2 轮

**第 2 轮：篇数上限询问**
```markdown
已选择发散模式（AI 自动多轮搜索）。
篇数上限是多少？（默认 30，最大 100）
```

**第 3 轮：发散轮数询问**
```markdown
发散几轮？（默认 3，范围 2-10）
```

**第 4 轮：配额预览 + 确认询问**
```markdown
配额分配预览：
- 总上限 50 篇，发散 3 轮
- 每轮配额 = 总上限 ÷ 轮数，余数加到第一轮
- 本次分配: [18, 16, 16]

确认按该参数开始发散搜索吗？
```

**第 5 轮：超链接格式选择**

用户确认参数后，询问超链接格式：

```markdown
报告中超链接格式：

| 选项 | 格式   | 说明                              |
|:----:|:------|:----------------------------------|
| A    | 纯文本  | 原帖引用为纯文本，无超链接（默认）     |
| B    | 超链接  | 原帖引用可直接点击跳转到小红书       |

回复 `A` 或 `B`（默认 A）。
```

**模式 B 交互序列总结**：
1. 模式选择确认
2. 篇数上限询问
3. 发散轮数询问
4. 配额预览 + 确认询问
5. 超链接格式选择

#### 5. 禁止合并问题

澄清阶段输出时，**禁止**以下行为：
- 将"篇数上限"和"发散轮数"合并到同一输出
- 将"配额预览"和"超链接格式"合并到同一输出
- 在展示衍生结果时同时询问超链接格式

**澄清阶段完成标志**：
- 模式 A：用户明确确认关键词 + 选择超链接格式，进入阶段二
- 模式 B：用户明确确认参数 + 选择超链接格式，进入阶段二

**[核心要求] 确认后必须立即创建搜索目录**：
- 用户确认关键词或发散参数后，**必须立即**在调用子 skill 前创建搜索目录
- 目录格式：`YYYYMMDD_HHmmSS_搜索主旨`
- **时间戳必须使用当前系统时间，禁止使用示例、模板或虚构时间**
- 后续所有产出文件必须在此目录下

### 阶段二：执行阶段（确认后进入）

此阶段编排各个子 skills 完成数据抓取和报告生成。

**编排层职责**：确保登录 → 抓取数据 → 生成报告 → 格式化报告 → 发送报告

#### 执行任务清单

进入阶段二后，必须立即将以下内容写入 `{OUTPUT_DIR}/tasks.md`：

```markdown
## 执行任务清单

- [ ] 确保登录
- [ ] 抓取数据
- [ ] 生成报告
- [ ] 格式化报告
- [ ] 发送报告
```

**核心要求**：
- 每完成一项任务，必须立即更新 tasks.md 将对应的 `[ ]` 改为 `[x]`
- 阶段二结束前，必须执行 `python scripts/verify_tasks.py <tasks_file_path>` 验证全部完成
- 验证失败必须报错中止，禁止发送未完成的报告

#### 登录轮询配置

`确保登录`步骤使用以下配置：

- `LOGIN_POLL_INTERVAL_SEC`：登录输出轮询间隔（秒）
  - 默认值：`2`
  - 建议范围：`1-10`
  - 若配置值非法（非数字、<=0 或超出建议范围），**必须**回退到默认值 `2`，并在日志中说明回退原因

#### 登录步骤执行模板（调用编排脚本）

```text
1) 读取 LOGIN_POLL_INTERVAL_SEC（默认 2；非法值回退到 2）
2) 调用：python .claude/skills/xiaohongshu-scraper/scripts/orchestrate_login.py --poll-interval <LOGIN_POLL_INTERVAL_SEC> --tasks-file <tasks_file_path>
3) 监听 orchestrate_login.py 输出：
   - 原样透传 login_xhs.py 输出，保留 COOKIE_FINGERPRINT 等排障信息
   - 重点关注 LOGIN_EVENT: {...} 结构化事件
4) 出现 NEED_LOGIN:<abs_path> 或 `LOGIN_EVENT: {"event": "NEED_LOGIN", ...}`：
   - 立即向用户发送：
     “请扫码登录小红书。二维码文件：<abs_path>”
5) 出现 LOGIN_OK / LOGIN_SUCCESS（或对应 LOGIN_EVENT）后：
   - orchestrate_login.py 会负责将 tasks.md 中“确保登录”标记为完成
   - 继续执行下一步
6) 出现 LOGIN_TIMEOUT / LOGIN_FAILED / ORCHESTRATOR_ERROR 后：
   - 报错并中止执行阶段
```

#### 发散模式执行约束

**[核心要求] 禁止提前预设关键词**：
- 发散模式下，Agent **禁止**在任意轮次开始前预设后续轮次的关键词
- 每个关键词**必须**在上一轮搜索完成后才能确定
- **禁止** Agent 在内心或笔记中规划好所有轮次的关键词

**[核心要求] 每轮必须输出发散决策报告**：
- 第 N 轮搜索完成后（N < 总轮数），**必须**输出”发散决策报告”
- 决策报告**必须**展示给用户

**发散决策报告格式**：

```markdown
### 第 N 轮发散决策

**本轮发现**：
- 高频品牌/主题：[发现 1、发现 2...]
- 争议焦点：[xxx]
- 信息盲区：[xxx]

**下一关键词**：xxx

**衍生理由**：基于 [具体发现]，为 [目的] 衍生出此关键词
```

**衍生理由示例**：
```
✓ 正确示例：
  基于发现”品牌A被高频提及且口碑较好”，为进一步了解”品牌A的具体优缺点”，衍生关键词”品牌A 评测”

✗ 错误示例：
  进一步了解品牌A（缺少因果链）
```

#### 发散模式执行流程

```
第 1 轮：
  → 基于用户原始主题生成初始关键词
  → 调用 fetch 抓取
  → 输出发散决策报告（包含本轮发现、下一关键词、衍生理由）

第 2 轮：
  → 使用决策报告中的关键词
  → 调用 fetch 抓取
  → 输出发散决策报告

...

最后一轮：
  → 使用决策报告中的关键词
  → 调用 fetch 抓取
  → 合并所有轮次数据为 raw.json
```

## 前置依赖配置

### Cookie 文件路径

`login_xhs.py` 与 `fetch_xhs.py` 不再读取 `XHS_AUTH_STATE` 环境变量。

两者固定使用同一份 Cookie 文件：

```text
.claude/skills/xiaohongshu-scraper/scripts/xhs_auth.json
```

实现约束：
- 路径始终基于 skill 脚本自身位置推导，不依赖当前工作目录
- `login_xhs.py` 写入后会输出 `COOKIE_FINGERPRINT[...]`，包含绝对路径、mtime、size、sha256
- `fetch_xhs.py` 读取前会输出同样口径的 `COOKIE_FINGERPRINT[...]`
- 若 login 与 fetch 的绝对路径 / 指纹不一致，应优先排查是否误用了不同文件
- 登录二维码截图固定输出到 `scripts/xhs_qr_login.png`，不会落到调用时 shell 的 cwd

### 依赖安装

```bash
pip install -r .claude/skills/xiaohongshu-scraper/scripts/requirements.txt
```

### Windows 环境

脚本会优先使用系统 Edge (`channel="msedge"`) 以减少风控特征。

### Linux 无屏幕环境配置（Xvfb 虚拟显示器）⚠️ 必须

**[核心要求]** 在 Linux/WSL 服务器（无物理显示器）上运行时，必须使用 Xvfb 创建虚拟屏幕。脚本强制使用有头模式以确保登录成功率，无 `DISPLAY` 环境变量时将报错退出。

```bash
# 安装 Xvfb 和依赖
sudo apt-get install -y xvfb libgbm1 libnss3 libatk-bridge2.0-0

# 启动虚拟显示器
Xvfb :99 -screen 0 1920x1080x24 &
export DISPLAY=:99
```

## 常见限制与故障排除

| 问题 | 原因 | 处理方式 |
|-----|------|---------|
| 检测到未登录 | Cookie 过期或未登录 | 执行 `scripts/orchestrate_login.py`，由其拉起 `login_xhs.py` 并在当前会话中完成扫码登录 |
| 风控封禁 | 小红书检测到自动化 | 脚本已内置延时，**禁止取消延时** |
| 元素选择器失效 | 小红书页面改版 | 检查 `scripts/xhs_selectors.py` 并更新选择器 |
| 二维码过期或超时 | 登录等待时间过长 | 使用 `--timeout N` 调整超时时间 |
