# 小红书报告格式化 Skill

对现有 `{主题}.md` 报告进行后处理，增强 emoji 使用，提升报告的趣味性和可读性。

---

## 🎯 使用场景

在报告草稿已经整理为 `{主题}.md` 后，可调用此 skill 进行格式美化：

```
1. 解析报告文件名（从 OUTPUT_DIR 路径解析主题）
2. 已有 {主题}.md
3. 调用 xiaohongshu-formatter
4. 输出格式增强后的 {主题}.md
```

### 报告文件名解析

**从 OUTPUT_DIR 路径解析主题名称**：

```
OUTPUT_DIR 格式: data/xiaohongshu/YYYYMMDD_HHmmSS_主题/
示例: data/xiaohongshu/20260312_150611_北京出发国外目的地推荐/

解析规则:
1. 取目录名最后一段: "20260312_150611_北京出发国外目的地推荐"
2. 按下划线分割，取第三段（索引 2）作为主题: "北京出发国外目的地推荐"
3. 报告文件名: {主题}.md

代码示例:
  import os
  dir_name = os.path.basename(output_dir.rstrip("/"))
  theme = dir_name.split("_", 2)[2] if "_" in dir_name else dir_name
  report_file = f"{theme}.md"
```

**fallback**：如果解析失败，使用默认文件名 `_index.md`。

## 📋 Emoji 使用规范（Moderate 风格）

### 原则
- **适度使用**：增强可读性和趣味性，不过度装饰
- **语义相关**：emoji 与内容相关，不随机堆砌
- **保持专业**：报告依然保持数据分析的可信度

### 必须使用 Emoji 的位置

| 位置 | Emoji 示例 | 说明 |
|:-----|:-----------|:-----|
| 报告标题 | ☕️ 北京咖啡厅探店报告 ☕️ | 前后各一个 |
| 8个主要板块标题 | 📊 搜索概览、🔥 声量分析、💬 高频共识 | 标题前加一个 |
| 数据来源说明标题 | 📍 数据来源说明 | 标题前加定位图标 |
| 场景推荐表格 | ☕ Dirty首选、📚 自习办公 | 场景列加 emoji |
| 避雷问题类型 | 🚫 空间问题、💸 价格问题 | 问题分类标识 |
| 优缺点列表前缀 | ✨ 核心优点、😤 核心痛点 | 列表标题 |
| 关键洞察编号后 | 1. ☕ Dirty是...、2. 🏙️ 胡同咖啡... | 每条洞察一个 |
| 表格内的评级 | ⭐⭐⭐⭐⭐ | 质量评分 |
| 热度指示 | 🔥🔥🔥 | 3-5个火焰表示热度 |

### 可选使用 Emoji 的位置

| 位置 | 使用建议 |
|:-----|:---------|
| 拥挤程度 | 😊/😌/😰 表达感受 |

### 不使用 Emoji 的位置

- 引用的用户评论内容（保持原始感）
- 来源标注行
- 长段落正文
- 价格档位（保持清晰简洁）

### 发散路径板块规则（新增）

当报告包含发散模式结果时，formatter 必须识别并处理发散路径板块：

- 识别标题：`## 🧭 搜索发散路径`（兼容无 emoji 标题 `## 搜索发散路径`）
- 仅处理该板块中的表格样式与可读性，不修改原始数据语义
- 保持列结构不变：`轮次 | 关键词 | 实际篇数 | 发散理由`
- 如果报告无该板块，必须跳过发散路径处理并继续其他板块

### 发散路径 Emoji 增强规则（新增）

对发散路径表格执行适度增强：

- 关键词列：可按语义补充前缀 emoji（如品牌探索、价格探索、场景探索）
- 发散理由列：可用轻量 emoji 强化可读性（如 💡 表示新发现、🔁 表示迭代延伸）
- `合计` 行保持简洁，不堆叠 emoji
- 禁止更改关键词和理由的核心文本含义；仅允许前后缀增强

## 🛠️ 执行流程

1. **读取现有报告**
   - 从 OUTPUT_DIR 解析主题名称（见上方说明）
   - 定位到搜索目录下的 `{主题}.md`
   - 读取全部内容
   - 检查是否存在 `id_url_map.json` 文件

2. **URL 替换（超链接功能）**

   如果 `id_url_map.json` 文件存在：

   - 读取 ID-URL 映射文件，获取所有有效的 post_id
   - 将报告中的所有 `(id:{post_id})` 占位符替换为帖子详情页 URL
   - 替换格式：`(id:{post_id})` → `(https://www.xiaohongshu.com/explore/{post_id})`
   - 映射文件用于验证 post_id 存在性，URL 使用 `explore` 格式构造

3. **应用格式化**
   - 按照上述 emoji 规范增强格式
   - 若存在「搜索发散路径」板块，按发散路径规则进行专门格式化
   - 确保各主要章节之间使用 Markdown 分割线（`---`）
   - 清理用户评论中的小红书格式标记
   - 保持原有内容和结构不变

4. **覆盖原文件**
   - 将格式化后的内容写回 `{主题}.md`
   - 或者先输出到终端供用户预览

## 📏 分割线规范

### 必须添加分割线的位置

在报告的以下主要章节之间**必须**添加 Markdown 分割线（`---`）：

| 位置 | 说明 |
|:-----|:-----|
| 搜索概览 → 品牌/主题声量分析 | 第一章节后 |
| 品牌/主题声量分析 → 评论区高频共识 | 第二章节后 |
| 评论区高频共识 → 场景化决策建议 | 第三章节后 |
| 场景化决策建议 → 价格/规格参考 | 第四章节后 |
| 价格/规格参考 → 避雷指南 | 第五章节后 |
| 避雷指南 → 关键洞察 | 第六章节后 |

**注意：** 子章节之间不需要分割线，只在主要板块之间使用。

## 🧹 清理规则

在格式化过程中，同时清理小红书评论系统的格式标记：

### 回复标记清理

小红书评论中，回复他人的评论会带有 `":"` 前缀，需要移除。

**规则：**
- 匹配模式：`_(":"` → 替换为 `_("`
- 只在用户评论（斜体引号包裹的内容）中应用
- 不影响其他位置的 `":"` 符号
