# 小红书爬虫执行规范

## 目的

定义小红书爬虫 skill 的执行规范，包括约束标签体系和优先级处理规则。

---

## 新增需求

### 需求:Agent 执行规范必须使用三层约束标签
SKILL.md 和 workflow.md 中的所有操作指令必须使用明确的优先级标签标识约束等级。

#### 场景:核心要求标签
- **当** 指令描述任务成功必须完成的步骤
- **那么** 必须使用 `[核心要求]` 标签
- **并且** 关键词使用"必须"、"禁止"、"绝对"

#### 场景:标准操作标签
- **当** 指令描述强烈建议但可在特殊情况下灵活调整的步骤
- **那么** 必须使用 `[标准操作]` 标签
- **并且** 关键词使用"应当"、"要求"

#### 场景:灵活选项标签
- **当** 指令描述可选的替代路径或特殊情况的备选方案
- **那么** 必须使用 `[灵活选项]` 标签
- **并且** 关键词使用"可"、"可以"、"可选"

#### 场景:参考信息标签
- **当** 指令仅提供提示信息，不约束 Agent 行为
- **那么** 必须使用 `[参考信息]` 标签
- **并且** 关键词使用"注意"、"提示"、⏱️

### 需求:必须定义优先级冲突处理规则
文档必须包含清晰的优先级定义章节，说明各标签的优先级关系和冲突处理逻辑。

#### 场景:优先级降序
- **当** Agent 面临多个冲突指令
- **那么** 必须按照 P0 > P1 > P2 > P3 的优先级降序执行

#### 场景:P0 受阻处理
- **当** P0 核心要求无法执行
- **那么** Agent 必须停止任务并报告失败

#### 场景:P1 降级到 P2
- **当** P1 标准操作受阻但存在 P2 灵活选项
- **那么** Agent 可降级使用 P2 方案
- **并且** 必须在输出中标注降级原因

#### 场景:P2/P3 忽略
- **当** P2 或 P3 指令无法执行
- **那么** Agent 可忽略该指令并继续任务

## 修改需求

### 需求:SKILL.md 必须移除模糊语言
SKILL.md 中所有使用"建议"、"推荐"等模糊词汇的指令必须替换为对应优先级的明确表述。

#### 场景:Xvfb 配置指令
- **当** SKILL.md 描述 Linux 环境使用 Xvfb
- **那么** 必须标记为 `[标准操作]` 应当使用 Xvfb
- **并且** headless 降级方案标记为 `[灵活选项]`

#### 场景:数据保存指令
- **当** SKILL.md 描述抓取结果保存
- **那么** 必须标记为 `[核心要求]` 必须保存到临时 JSON

#### 场景:时间提示信息
- **当** SKILL.md 提示抓取过程耗时
- **那么** 必须标记为 `[参考信息]` ⏱️ 过程可能需要几分钟

#### 场景:故障排除指令
- **当** SKILL.md 描述元素选择器失效时的处理
- **那么** 必须标记为 `[核心要求]` 必须检查 scripts/selectors.py

### 需求:workflow.md 必须同步约束标签
workflow.md 中的步骤描述必须使用与 SKILL.md 一致的优先级标签体系。

#### 场景:强制输出流程步骤
- **当** workflow.md 步骤 4 描述输出要求
- **那么** 每个子步骤必须添加 `[核心要求]` 标签前缀
- **并且** 保持"必须"、"禁止"等关键词

#### 场景:新增分层步骤
- **当** workflow.md 添加标准操作和灵活选项步骤
- **那么** 必须使用对应的 `[标准操作]` 和 `[灵活选项]` 标签

## 新增需求 (来自 remove-xhs-post-navigation)

### 需求: 抓取数据必须不包含 URL 字段
`fetch_xhs.py` 脚本输出的 JSON 数据必须不包含 `url` 字段，只保留标题、内容、作者、日期、点赞数、收藏数、评论数和评论列表。

#### 场景: 数据抓取
- **当** 脚本抓取小红书帖子数据
- **那么** 输出的 JSON 必须包含 8 个字段：title, content, author, date, likes, collects, comments_count, comments
- **并且** 禁止包含 url 字段

#### 场景: 数据兼容性
- **当** 读取已有的 raw.json 数据
- **那么** Agent 应当正常处理，不依赖 url 字段的存在

## 新增需求 (来自 standardize-xhs-clarification-output-format)

### 需求:确认后必须立即创建搜索目录
系统必须在用户确认关键词后立即创建搜索目录，所有后续产出文件必须在此目录下，避免搜索阶段出现bug时重复创建文件夹。

#### 场景:目录创建时机
- **当** 用户在澄清阶段明确确认搜索关键词
- **那么** 系统必须在调用 fetch_xhs.py 之前立即创建搜索目录
- **禁止** 等待爬虫脚本执行完成后再创建目录

#### 场景:目录命名格式
- **当** 创建搜索目录
- **那么** 目录格式必须为 `YYYYMMDD_HHmmSS_搜索主旨`（使用占位符）
- **那么** 时间戳部分（`YYYYMMDD_HHmmSS`）必须替换为当前系统时间
- **那么** 搜索主旨必须基于用户确认的关键词主题提取
- **示例**：`20260303_182300_北京粤菜推荐`（18:23 执行时使用 182300）
- **禁止** 使用示例时间或虚构时间

#### 场景:文件输出约束
- **当** 调用 fetch_xhs.py 脚本
- **那么** 必须使用 --output 参数指定输出到搜索目录下的 raw.json
- **当** 生成 _index.md 报告
- **那么** 必须输出到搜索目录下
- **禁止** 在搜索目录外创建任何产出文件

#### 场景:目录冲突防护
- **当** 搜索目录已存在
- **那么** 系统必须警告用户并询问是否覆盖或使用新目录
- **或者** 自动在目录名后添加序号后缀（如 _1、_2）避免冲突

### 需求: 目录时间戳必须使用当前系统时间

Agent 在创建搜索目录时，时间戳必须使用当前系统时间，禁止使用示例、模板或虚构时间。

#### 场景: 目录创建时间要求

- **当** Agent 创建搜索目录
- **那么** 必须使用当前系统时间生成时间戳
- **并且** 禁止使用示例时间（如 `143000`）或任何虚构时间
- **示例**：`20260303_182300_北京粤菜推荐`（18:23 执行时）

## 新增需求 (来自 improve-xhs-skill-ai-only-analysis)

### 需求: Agent 禁止使用语义分析脚本

SKILL.md 中的核心要求清单**必须**包含以下约束：生成 `_index.md` 时，禁止调用任何外部脚本进行**文本分析或内容提取**（包括但不限于：Python `re` 正则匹配、`collections.Counter` 词频统计、实体识别、情感分析脚本等）。所有关于内容的理解、分析、总结（如品牌识别、优缺点提炼、情感分析、观点归纳）必须基于 LLM 的自然语言理解能力完成。允许基本数值统计（如帖子数量、评论总数、数值计算）。

#### 场景: 核心要求更新

- **当** SKILL.md 被更新
- **那么** 核心要求清单**必须**包含上述禁止语义分析脚本的约束

### 需求: Emoji 使用自由化

Agent 在生成报告时使用的 emoji **不**受固定映射规则约束。

#### 场景: Emoji 自由选择

- **当** Agent 生成报告时
- **那么** Agent 根据上下文自行判断使用何种 emoji 及其位置
- **并且** **无需**遵循任何固定的 emoji 映射规则

### 需求: 报告必须使用 Emoji 增强可读性

Agent 在生成 `_index.md` 报告时，必须在报告标题和 8 个主要板块使用 emoji 增强可读性。

#### 场景: 报告标题 Emoji 要求

- **当** Agent 生成报告标题
- **那么** 必须使用适当的 emoji
- **并且** emoji 应当与报告主题相关

#### 场景: 报告板块 Emoji 要求

- **当** Agent 生成报告的 8 个主要板块
- **那么** 每个板块标题必须使用 emoji
- **并且** 具体使用的 emoji 由 Agent 根据内容上下文自行判断

#### 场景: 必须使用 Emoji 的 8 个板块

- **当** Agent 生成 `_index.md` 报告
- **那么** 以下位置必须使用 emoji：
  1. 报告标题
  2. 搜索概览
  3. 品牌/主题声量分析
  4. 评论区高频共识
  5. 场景化决策建议
  6. 价格/规格参考
  7. 避雷指南
  8. 关键洞察

### 需求: 定性描述替代精确计数

报告中的提及次数**应当**使用定性描述而非精确计数。

#### 场景: 使用定性描述

- **当** Agent 需要描述某个观点的提及频率
- **那么** Agent **应当**使用定性描述（如"多位用户提到"、"广泛讨论"、"高频反馈"）
- **并且** **避免**使用"提及 ~X 次"这种精确计数格式

## 移除需求 (来自 improve-xhs-skill-ai-only-analysis)

### 需求: 移除 Python 代码示例

**Reason**: Python 代码示例会诱导 Agent 模仿使用脚本处理数据的模式，违反禁止语义分析脚本的核心要求。

**Migration**: 用自然语言描述替代代码示例。

#### 场景: 移除数据读取代码示例

- **当** 更新 SKILL.md 中的"数据读取"段落
- **那么** **删除** `sum(post["comments_count"] for post in posts)` 代码示例
- **并且** 改用自然语言描述："累加所有帖子的 `comments_count` 字段"

#### 场景: 移除 Emoji 映射函数

- **当** 更新 SKILL.md 中的 Emoji 相关内容
- **那么** **删除** `map_intent_emoji()` Python 函数定义
- **并且** 改为鼓励性描述："报告应适当使用 emoji 增强可读性，具体 emoji 及其位置由 Agent 根据内容上下文自行判断"

### 需求: 移除固定 Emoji 映射表格

**Reason**: 固定映射限制了 Agent 的灵活性，且不需要此类约束。

**Migration**: 删除模板中的固定 emoji 和映射规则说明。

#### 场景: 移除搜索策略表格中的固定 emoji

- **当** 更新报告模板中的"搜索策略"表格
- **那么** **移除**关键词列前的固定 emoji（如 📍、🏪、⚠️）
- **并且** **删除**表格下方的"Emoji 映射规则"说明

#### 场景: 移除模板标题中的固定 emoji

- **当** 更新报告模板
- **那么** **移除**各级标题前的固定 emoji
- **并且** 让 Agent 自行决定是否使用 emoji

### 需求: 移除精确计数格式

**Reason**: "提及 ~X 次"格式诱导 Agent 进行词频统计。

**Migration**: 使用定性描述替代。

#### 场景: 核心优点/痛点格式调整

- **当** 更新"核心优点"和"核心痛点"模板
- **那么** **删除**"提及 ~X 次"格式
- **并且** 使用定性描述："多位用户提到"、"广泛讨论"、"高频反馈"等

### 需求: 移除引用格式中的 Emoji 映射规则

**Reason**: 不再需要固定的 emoji 映射规则。

**Migration**: 删除相关说明段落，添加数据来源说明。

#### 场景: 清理模板底部说明

- **当** 更新模板底部的"引用格式说明"
- **那么** **删除**"Emoji 映射规则"相关内容
- **并且** 添加数据来源说明："所有内容分析必须基于 raw.json 中的原始数据，使用 LLM 自然语言理解能力进行归纳总结，禁止使用自动化脚本进行词频统计。"
