# 小红书爬虫执行规范

## 目的

定义小红书爬虫 skill 的执行规范，包括约束标签体系和优先级处理规则。

---

## 新增需求

### 需求:Agent 执行规范必须使用三层约束标签
SKILL.md 和 workflow.md 中的所有操作指令必须使用明确的优先级标签标识约束等级。

#### 场景:核心要求标签
- **当** 指令描述任务成功必须完成的步骤
- **那么** 必须使用 `[核心要求]` 标签
- **并且** 关键词使用"必须"、"禁止"、"绝对"

#### 场景:标准操作标签
- **当** 指令描述强烈建议但可在特殊情况下灵活调整的步骤
- **那么** 必须使用 `[标准操作]` 标签
- **并且** 关键词使用"应当"、"要求"

#### 场景:灵活选项标签
- **当** 指令描述可选的替代路径或特殊情况的备选方案
- **那么** 必须使用 `[灵活选项]` 标签
- **并且** 关键词使用"可"、"可以"、"可选"

#### 场景:参考信息标签
- **当** 指令仅提供提示信息，不约束 Agent 行为
- **那么** 必须使用 `[参考信息]` 标签
- **并且** 关键词使用"注意"、"提示"、⏱️

### 需求:必须定义优先级冲突处理规则
文档必须包含清晰的优先级定义章节，说明各标签的优先级关系和冲突处理逻辑。

#### 场景:优先级降序
- **当** Agent 面临多个冲突指令
- **那么** 必须按照 P0 > P1 > P2 > P3 的优先级降序执行

#### 场景:P0 受阻处理
- **当** P0 核心要求无法执行
- **那么** Agent 必须停止任务并报告失败

#### 场景:P1 降级到 P2
- **当** P1 标准操作受阻但存在 P2 灵活选项
- **那么** Agent 可降级使用 P2 方案
- **并且** 必须在输出中标注降级原因

#### 场景:P2/P3 忽略
- **当** P2 或 P3 指令无法执行
- **那么** Agent 可忽略该指令并继续任务

SKILL.md 中所有使用"建议"、"推荐"等模糊词汇的指令必须替换为对应优先级的明确表述。

- **那么** 必须标记为 `[标准操作]` 应当使用 Xvfb

#### 场景:新增分层步骤

### 需求: 抓取数据必须不包含 URL 字段
`fetch_xhs.py` 脚本输出的 JSON 数据必须不包含 `url` 字段，只保留标题、内容、作者、日期、点赞数、收藏数、评论数和评论列表。

#### 场景: 数据抓取
- **当** 脚本抓取小红书帖子数据
- **那么** 输出的 JSON 必须包含 8 个字段：title, content, author, date, likes, collects, comments_count, comments
- **并且** 禁止包含 url 字段

#### 场景: 数据兼容性
- **当** 读取已有的 raw.json 数据
- **那么** Agent 应当正常处理，不依赖 url 字段的存在

## 新增需求 (来自 standardize-xhs-clarification-output-format)

### 需求:确认后必须立即创建搜索目录
系统必须在用户确认关键词后立即创建搜索目录，所有后续产出文件必须在此目录下，避免搜索阶段出现bug时重复创建文件夹。

#### 场景:目录创建时机
- **当** 用户在澄清阶段明确确认搜索关键词
- **那么** 系统必须在调用 fetch_xhs.py 之前立即创建搜索目录
- **禁止** 等待爬虫脚本执行完成后再创建目录

#### 场景:目录命名格式
- **当** 创建搜索目录
- **那么** 目录格式必须为 `YYYYMMDD_HHmmSS_搜索主旨`（使用占位符）
- **那么** 时间戳部分（`YYYYMMDD_HHmmSS`）必须替换为当前系统时间
- **那么** 搜索主旨必须基于用户确认的关键词主题提取
- **示例**：`20260303_182300_北京粤菜推荐`（18:23 执行时使用 182300）
- **禁止** 使用示例时间或虚构时间

#### 场景:文件输出约束
- **当** 调用 fetch_xhs.py 脚本
- **那么** 必须使用 --output 参数指定输出到搜索目录下的 raw.json
- **当** 生成 _index.md 报告
- **那么** 必须输出到搜索目录下
- **禁止** 在搜索目录外创建任何产出文件

#### 场景:目录冲突防护
- **当** 搜索目录已存在
- **那么** 系统必须警告用户并询问是否覆盖或使用新目录
- **或者** 自动在目录名后添加序号后缀（如 _1、_2）避免冲突

## 新增需求 (来自 improve-xhs-skill-ai-only-analysis)

### 需求: Agent 禁止使用语义分析脚本

SKILL.md 中的核心要求清单**必须**包含以下约束：生成 `_index.md` 时，禁止调用任何外部脚本进行**文本分析或内容提取**（包括但不限于：Python `re` 正则匹配、`collections.Counter` 词频统计、实体识别、情感分析脚本等）。所有关于内容的理解、分析、总结（如品牌识别、优缺点提炼、情感分析、观点归纳）必须基于 LLM 的自然语言理解能力完成。允许基本数值统计（如帖子数量、评论总数、数值计算）。

#### 场景: 核心要求更新

- **当** SKILL.md 被更新
- **那么** 核心要求清单**必须**包含上述禁止语义分析脚本的约束

### 需求: Emoji 使用自由化

Agent 在生成报告时使用的 emoji **不**受固定映射规则约束。

#### 场景: Emoji 自由选择

- **当** Agent 生成报告时
- **那么** Agent 根据上下文自行判断使用何种 emoji 及其位置
- **并且** **无需**遵循任何固定的 emoji 映射规则

### 需求: 报告必须使用 Emoji 增强可读性

Agent 在生成 `_index.md` 报告时，必须在报告标题和 8 个主要板块使用 emoji 增强可读性。

#### 场景: 报告标题 Emoji 要求

- **当** Agent 生成报告标题
- **那么** 必须使用适当的 emoji
- **并且** emoji 应当与报告主题相关

#### 场景: 报告板块 Emoji 要求

- **当** Agent 生成报告的 8 个主要板块
- **那么** 每个板块标题必须使用 emoji
- **并且** 具体使用的 emoji 由 Agent 根据内容上下文自行判断

#### 场景: 必须使用 Emoji 的 8 个板块

- **当** Agent 生成 `_index.md` 报告
- **那么** 以下位置必须使用 emoji：
  1. 报告标题
  2. 搜索概览
  3. 品牌/主题声量分析
  4. 评论区高频共识
  5. 场景化决策建议
  6. 价格/规格参考
  7. 避雷指南
  8. 关键洞察

### 需求: 定性描述替代精确计数

报告中的提及次数**应当**使用定性描述而非精确计数。

#### 场景: 使用定性描述

- **当** Agent 需要描述某个观点的提及频率
- **那么** Agent **应当**使用定性描述（如"多位用户提到"、"广泛讨论"、"高频反馈"）
- **并且** **避免**使用"提及 ~X 次"这种精确计数格式

## 新增需求 (来自 add-xhs-divergence-search)

### 需求: 发散模式执行循环
Agent 在发散模式下必须执行迭代式搜索循环：每轮搜索 → 分析结果 → 决定下一个关键词 → 重复。

#### 场景: 初始关键词生成
- **当** 进入发散模式执行阶段（第 1 轮）
- **那么** Agent 必须基于用户原始主题生成 1 个初始搜索关键词
- **并且** 关键词应当直接反映用户主题的核心含义

#### 场景: 单轮搜索执行
- **当** Agent 执行第 N 轮搜索
- **那么** 必须调用 `fetch_xhs.py`，传入：
  - `--keywords` 为本轮关键词（仅 1 个）
  - `--max-posts` 为本轮配额
  - `--output` 为 `{search_dir}/raw_round_{N}.json`
  - `--seen-ids` 为 `{search_dir}/seen_ids.txt`
- **并且** 必须等待本轮搜索完成后再进入分析阶段

#### 场景: AI 分析与关键词决策
- **当** 第 N 轮搜索完成且 N < 总轮数
- **那么** Agent 必须阅读本轮及之前所有轮次的搜索结果
- **那么** Agent 必须基于 LLM 自然语言理解能力决定下一个关键词
- **那么** Agent 必须记录本次决策的理由（用于发散路径板块）
- **并且** 下一个关键词**必须与用户原始主题保持关联**
- **并且** 应当基于已搜索结果中发现的新方向（如高频品牌、争议点、信息盲区）
- **并且** 应当避免与已搜索过的关键词高度重复

#### 场景: 循环终止
- **当** 当前轮次 = 总轮数（最后一轮搜索完成）
- **那么** Agent 必须停止搜索循环
- **那么** 必须合并所有轮次的数据为最终 `raw.json`

### 需求: 配额平均分配
系统必须将篇数上限均匀分配到每轮搜索中。

#### 场景: 均匀分配
- **当** 篇数上限可被轮数整除
- **那么** 每轮配额 = 篇数上限 ÷ 轮数
- **示例**：50 篇 ÷ 5 轮 = 每轮 10 篇

#### 场景: 不可整除时的分配
- **当** 篇数上限不可被轮数整除
- **那么** 余数必须加到第一轮
- **示例**：50 篇 ÷ 3 轮 = [18, 16, 16]

### 需求: 跨轮去重
`fetch_xhs.py` 必须支持 `--seen-ids` 参数实现跨调用去重。

#### 场景: seen-ids 参数
- **当** `fetch_xhs.py` 接收到 `--seen-ids <filepath>` 参数
- **那么** 启动时必须读取文件中已有的 note_id 集合
- **并且** 搜索时必须跳过已见的 note_id
- **并且** 完成后必须将本次新发现的 note_id 追加写入该文件

#### 场景: seen-ids 文件格式
- **当** 读写 seen-ids 文件
- **那么** 文件格式必须为每行一个 note_id（纯文本）
- **并且** 文件不存在时必须自动创建

#### 场景: 去重后篇数不足
- **当** 去重后实际抓取篇数少于本轮配额
- **那么** 系统必须接受实际篇数，不进行补量
- **并且** 在最终报告中标注实际篇数

### 需求: 数据合并
Agent 必须在所有轮次完成后将分轮数据合并为最终 raw.json。

#### 场景: 合并格式
- **当** 所有轮次搜索完成
- **那么** Agent 必须将所有 `raw_round_N.json` 中的帖子合并为一个 `raw.json`
- **并且** `raw.json` 必须包含 `divergence_path` 字段（发散路径数据）
- **并且** `raw.json` 的 `search_strategy` 字段由发散路径数据替代
- **并且** 必须保留 `search_time`、`keywords`（所有轮次关键词合集）、`posts` 字段

### 需求: 发散模式进度输出
Agent 在发散搜索过程中必须输出进度信息。

#### 场景: 轮次进度
- **当** 开始新一轮搜索
- **那么** 应当输出：`[发散轮次 N/R] 关键词: xxx (配额 M 篇)`
- **当** 一轮搜索完成并决定下一个关键词
- **那么** 应当输出：`[发散决策] 下一关键词: xxx | 理由: xxx`

  - `--seen-ids` 为 `{search_dir}/seen_ids.txt`
- **并且** 必须等待本轮搜索完成后再进入分析阶段

#### 场景: AI 分析与关键词决策
- **当** 第 N 轮搜索完成且 N < 总轮数
- **那么** Agent 必须阅读本轮及之前所有轮次的搜索结果
- **那么** Agent 必须基于 LLM 自然语言理解能力决定下一个关键词
- **那么** Agent 必须记录本次决策的理由（用于发散路径板块）
- **并且** 下一个关键词**必须与用户原始主题保持关联**
- **并且** 应当基于已搜索结果中发现的新方向（如高频品牌、争议点、信息盲区）
- **并且** 应当避免与已搜索过的关键词高度重复

#### 场景: 循环终止
- **当** 当前轮次 = 总轮数（最后一轮搜索完成）
- **那么** Agent 必须停止搜索循环
- **那么** 必须合并所有轮次的数据为最终 `raw.json`

### 需求: 配额平均分配
系统必须将篇数上限均匀分配到每轮搜索中。

#### 场景: 均匀分配
- **当** 篇数上限可被轮数整除
- **那么** 每轮配额 = 篇数上限 ÷ 轮数
- **示例**：50 篇 ÷ 5 轮 = 每轮 10 篇

#### 场景: 不可整除时的分配
- **当** 篇数上限不可被轮数整除
- **那么** 余数必须加到第一轮
- **示例**：50 篇 ÷ 3 轮 = [18, 16, 16]

### 需求: 跨轮去重
`fetch_xhs.py` 必须支持 `--seen-ids` 参数实现跨调用去重。

#### 场景: seen-ids 参数
- **当** `fetch_xhs.py` 接收到 `--seen-ids <filepath>` 参数
- **那么** 启动时必须读取文件中已有的 note_id 集合
- **并且** 搜索时必须跳过已见的 note_id
- **并且** 完成后必须将本次新发现的 note_id 追加写入该文件

#### 场景: seen-ids 文件格式
- **当** 读写 seen-ids 文件
- **那么** 文件格式必须为每行一个 note_id（纯文本）
- **并且** 文件不存在时必须自动创建

#### 场景: 去重后篇数不足
- **当** 去重后实际抓取篇数少于本轮配额
- **那么** 系统必须接受实际篇数，不进行补量
- **并且** 在最终报告中标注实际篇数

### 需求: 数据合并
Agent 必须在所有轮次完成后将分轮数据合并为最终 raw.json。

#### 场景: 合并格式
- **当** 所有轮次搜索完成
- **那么** Agent 必须将所有 `raw_round_N.json` 中的帖子合并为一个 `raw.json`
- **并且** `raw.json` 必须包含 `divergence_path` 字段（发散路径数据）
- **并且** `raw.json` 的 `search_strategy` 字段由发散路径数据替代
- **并且** 必须保留 `search_time`、`keywords`（所有轮次关键词合集）、`posts` 字段

### 需求: 发散模式进度输出
Agent 在发散搜索过程中必须输出进度信息。

#### 场景: 轮次进度
- **当** 开始新一轮搜索
- **那么** 应当输出：`[发散轮次 N/R] 关键词: xxx (配额 M 篇)`
- **当** 一轮搜索完成并决定下一个关键词
- **那么** 应当输出：`[发散决策] 下一关键词: xxx | 理由: xxx`

`login_xhs.py` 的 `_is_logged_in()` 方法必须验证搜索结果区域有帖子卡片存在，禁止仅用"无登录元素"判断登录成功。

- **那么** 必须返回 False

- **禁止** 仅通过 auth 文件是否存在判断登录状态
- **禁止** 仅通过是否能打开小红书页面判断登录状态
