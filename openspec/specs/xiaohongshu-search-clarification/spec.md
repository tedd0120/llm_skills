# 小红书搜索关键词澄清规范

## 目的

定义小红书爬虫 skill 搜索关键词澄清阶段的行为规范，确保搜索词符合用户意图。

---

## 新增需求

### 需求:用户输入必须作为整体处理
系统必须将用户输入的搜索建议视为一个整体文本，忽略内部所有分隔符（逗号、空格、分号等），AI 需要理解文本的主旨而非机械分割。

#### 场景:包含逗号的输入
- **当** 用户输入 "广州装修公司, 对比, 避坑"
- **那么** 系统必须将整个输入理解为"广州装修公司相关"的建议
- **那么** AI 必须基于整体主旨衍生关键词

#### 场景:自然语言输入
- **当** 用户输入 "我想了解新手买电脑需要注意什么"
- **那么** 系统必须识别主题为"新手买电脑"
- **那么** AI 必须基于"新手"和"买电脑"两个要素衍生关键词

### 需求:必须衍生恰好三个搜索词
系统必须基于用户建议的主旨衍生恰好 3 个搜索关键词，不得多于 3 个，不得少于 3 个。

#### 场景:标准衍生
- **当** 用户输入 "装修公司"
- **那么** 系统必须衍生恰好 3 个关键词
- **例如** ["装修公司推荐", "装修避坑", "装修预算"]

#### 场景:具体输入的衍生
- **当** 用户输入 "广州天河区哪家装修公司好"
- **那么** 系统必须衍生 3 个有关联但角度不同的关键词
- **例如** ["广州天河装修公司", "装修公司评价", "装修公司对比"]

### 需求:衍生关键词必须符合用户主旨
系统衍生的所有搜索词必须符合用户输入建议的核心意图和主旨，不得偏离用户意图。

#### 场景:地域保留
- **当** 用户输入包含地域信息 "广州装修"
- **那么** 衍生的关键词必须保留地域要素
- **反例** 不能衍生为"装修公司推荐"（丢失了"广州"）

#### 场景:场景保留
- **当** 用户输入包含场景信息 "新手买电脑"
- **那么** 衍生的关键词必须考虑"新手"场景
- **反例** 不能衍生为"高端游戏电脑推荐"（偏离了"新手"场景）

### 需求:必须执行澄清阶段
系统必须在首次调用命令时执行澄清阶段，与用户进行多轮问答确认搜索关键词，确认前不得执行爬虫操作。

#### 场景:首次调用流程
- **当** 用户首次调用 xiaohongshu-scraper 命令
- **那么** 系统必须解析用户输入
- **那么** 系统必须衍生 3 个关键词
- **那么** 系统必须以格式C展示给用户
- **那么** 系统必须等待用户反馈
- **禁止** 在确认前调用 fetch_xhs.py

#### 场景:多轮澄清
- **当** 用户要求修改关键词
- **那么** 系统必须根据用户反馈更新关键词
- **那么** 系统必须重新展示更新后的关键词
- **那么** 系统必须再次请求确认

### 需求:必须使用格式C展示关键词
系统必须使用格式C向用户展示衍生关键词，包含关键词列表、预计篇数和搜索理由。

#### 场景:格式C展示
- **当** 系统向用户展示衍生关键词
- **那么** 必须使用 Markdown 表格格式，包含以下列：序号、关键词、预计篇数、搜索理由
- **并且** 最后一行必须为合并列的总计行，格式为：`| **总计** | — | **N篇** | 默认上限 N 篇 |`
- **并且** 表格行数必须动态调整：行数 = 关键词数量 K + 1（K 范围为 1-10）
- **并且** 序号列必须居中对齐，关键词列左对齐，预计篇数列居中对齐，搜索理由列左对齐
- **并且** 每个关键词必须附带搜索理由，说明该关键词的搜索目的
- **示例** (K=3 时)：
  ```markdown
  我基于你的建议衍生了以下 3 个搜索词：

  | 序号 | 关键词 | 预计篇数 | 搜索理由 |
  |:----:|:------|:--------:|:---------|
  | 1 | [关键词一] | 10篇 | [理由一] |
  | 2 | [关键词二] | 10篇 | [理由二] |
  | 3 | [关键词三] | 10篇 | [理由三] |
  | **总计** | — | **30篇** | 默认上限 30 篇 |

  你可以：
  • 修改关键词数量 (如: "增加到5个" 或 "减少到2个"，范围1-10个)
  • 修改单个关键词 (如: "把第1个改成国产电视推荐")
  • 重新生成关键词 (如: "换一批关键词")
  • 提出关键词建议 (如: "加上'小米电视'这个品牌")
  • 修改搜索上限 (如: "增加到50篇" 或 "减少到20篇")
  • 确认开始爬取

  回复你的选择：
  ```

#### 场景:动态调整关键词数量
- **当** 用户要求增加关键词数量（如 "增加到5个"）
- **那么** 表格必须扩展为对应行数（6行：5个关键词 + 1个总计行）
- **当** 用户要求减少关键词数量（如 "减少到2个"）
- **那么** 表格必须收缩为对应行数（3行：2个关键词 + 1个总计行）

#### 场景:搜索理由填写
- **当** 填写搜索理由列
- **那么** 直译角度关键词的理由格式为："获取[主题]的整体推荐趋势"
- **那么** 对比角度关键词的理由格式为："获取[主题]的对比评价和用户体验"
- **那么** 避雷角度关键词的理由格式为："了解[主题]的常见问题和避坑指南"

### 需求:必须主动请求确认
系统必须在每次展示关键词后主动询问用户是否确认开始爬取，不得依赖模糊信号判断。

#### 场景:明确确认请求
- **当** 系统展示衍生关键词后
- **那么** 必须明确询问 "确认开始爬取吗？"
- **禁止** 仅依赖用户说"行"、"可以"等模糊表达

#### 场景:用户明确确认
- **当** 用户回复 "确认"、"开始"、"好的"、"就这样"
- **那么** 系统必须进入执行阶段
- **那么** 系统必须调用 fetch_xhs.py

### 需求:确认后必须执行爬虫
系统必须在用户确认关键词后执行爬虫操作，调用 fetch_xhs.py 传入确认的 3 个关键词。

#### 场景:执行爬虫
- **当** 用户确认搜索关键词
- **那么** 系统必须调用 fetch_xhs.py
- **那么** 系统必须传入确认的 3 个关键词作为 --keywords 参数
- **那么** 系统必须继续执行报告生成流程

### 需求:SKILL.md必须明确标记阶段
SKILL.md 文档必须使用明确的章节标记区分"澄清阶段"和"执行阶段"，确保 AI 能够识别当前所处阶段。

#### 场景:阶段标记
- **当** AI 阅读 SKILL.md
- **那么** 必须能够清晰识别"阶段一：澄清阶段"和"阶段二：执行阶段"
- **那么** 必须在澄清阶段完成后才进入执行阶段

### 需求:建议多角度衍生策略
系统建议（但不强制）从不同角度衍生关键词，包括：直译/直接搜索、对比/评价、避雷/负面等角度。

#### 场景:产品推荐类输入
- **当** 用户输入关于产品推荐的搜索建议
- **那么** 建议从以下角度衍生：
  - 直译角度：直接相关的搜索词
  - 对比角度：获取对比/评价信息
  - 避雷角度：获取风险/问题信息

#### 场景:技术问题类输入
- **当** 用户输入关于技术问题的搜索建议
- **那么** 建议从以下角度衍生：
  - 问题分类：不同类型的相关问题
  - 解决方案：不同解决途径
  - 同义词：相似表述的关键词

### 需求:确认后必须立即创建搜索目录
系统必须在用户确认关键词后立即创建搜索目录，所有后续产出文件必须在此目录下。

#### 场景:目录创建时机
- **当** 用户明确确认搜索关键词（回复"确认"、"开始"、"好的"、"就这样"等）
- **那么** 系统必须在调用 fetch_xhs.py 之前立即创建搜索目录
- **禁止** 在爬虫脚本执行后再创建目录

#### 场景:目录命名格式
- **当** 创建搜索目录
- **那么** 目录格式必须为 `yyyyMMdd_hhMMss_搜索主旨`
- **那么** 时间戳必须使用当前时间，精确到秒
- **那么** 搜索主旨必须基于用户确认的关键词主题提取
- **示例**：`20260303_143000_北大口腔拔牙`

#### 场景:文件输出约束
- **当** 生成输出文件（raw.json、_index.md）
- **那么** 所有文件必须输出到搜索目录内
- **禁止** 在搜索目录外创建任何产出文件

## 新增需求 (来自 add-xhs-divergence-search)

### 需求: 澄清阶段必须提供模式选择
系统在解析用户输入后、衍生关键词之前，必须让用户选择搜索模式。

#### 场景: 展示模式选择
- **当** 用户首次调用 xiaohongshu-scraper 命令
- **那么** 系统必须在解析输入主旨后展示两种模式选项：
  - A. 固定关键词模式（现有流程）
  - B. 发散模式（AI 自动发散搜索）
- **并且** 必须简要说明两种模式的区别

#### 场景: 用户选择固定关键词模式
- **当** 用户选择模式 A
- **那么** 系统必须按照现有澄清流程执行（衍生关键词 → 展示 → 确认）
- **并且** 后续流程与当前完全一致

#### 场景: 用户选择发散模式
- **当** 用户选择模式 B
- **那么** 系统必须进入发散模式参数设定阶段
- **禁止** 执行固定关键词的衍生和展示流程

### 需求: 发散模式必须收集参数
用户选择发散模式后，系统必须收集篇数上限和发散轮数两个参数。

#### 场景: 参数收集
- **当** 用户选择发散模式
- **那么** 系统必须询问用户设定以下参数：
  - 篇数上限（默认 30，硬上限 100）
  - 发散轮数（默认 3，范围 2-10）
- **并且** 必须展示配额分配预览（如 "50篇 ÷ 5轮 = 每轮 10 篇"）

#### 场景: 使用默认参数
- **当** 用户未指定参数而直接确认
- **那么** 系统必须使用默认值：30 篇、3 轮

#### 场景: 参数确认
- **当** 用户确认参数
- **那么** 系统必须立即创建搜索目录（与固定模式相同的命名规则）
- **并且** 进入发散模式执行阶段
