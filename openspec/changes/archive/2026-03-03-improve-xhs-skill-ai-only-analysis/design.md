## 上下文

小红书技能的 `SKILL.md` 是一个指令文档，指导 Agent 如何完成内容抓取和报告生成。当前文档包含一些 Python 代码示例（如 `sum(post["comments_count"]...)`、`map_intent_emoji()` 函数），这些示例可能误导 Agent 认为使用脚本处理数据是标准做法。

同时，报告模板中包含固定的 emoji 映射规则和"提及 ~X 次"的计数格式，这诱导 Agent 进行机械的词频统计而非语义理解。

## 目标 / 非目标

**目标：**
- 约束 Agent 在生成报告时使用 LLM 自然语言理解能力
- 移除可能诱导 Agent 使用脚本的内容
- 给予 Agent 更多 emoji 使用的自由度
- 提升报告的可读性和语义准确性

**非目标：**
- 不修改 `fetch_xhs.py` 脚本（爬虫功能保持不变）
- 不改变报告的基本结构和板块
- 不禁止基本数值统计（如 len、sum）

## 决策

### 决策 1: 添加 [核心要求] 约束

**选择：** 在 SKILL.md 的核心要求清单中添加明确的禁止条款

**理由：**
- [核心要求] 具有最高优先级，违反会导致任务失败
- Agent 会严格遵守核心要求
- 明确列出禁止的具体行为（re、Counter 等）避免歧义

**替代方案：**
- 仅在步骤描述中提及 → 约束力不足，Agent 可能忽略
- 创建单独的"禁止事项"章节 → 不如放在核心要求中突出

### 决策 2: 移除所有 Python 代码示例

**选择：** 删除 SKILL.md 中的 Python 代码片段和函数定义

**理由：**
- 代码示例会诱导 Agent 模仿这种模式
- 用自然语言描述同样能传达意图

**替代方案：**
- 保留代码但添加"仅供参考"注释 → Agent 仍可能模仿
- 改用伪代码 → 仍有诱导性

### 决策 3: Emoji 由 Agent 自行判断

**选择：** 移除固定映射表格和模板中的固定 emoji

**理由：**
- 固定映射限制了 Agent 的灵活性
- 不同上下文可能需要不同的 emoji
- LLM 完全有能力判断合适的 emoji

**替代方案：**
- 保留映射但标记为"建议" → Agent 可能过度依赖
- 提供更大范围的 emoji 库 → 增加复杂度

### 决策 4: 允许基本数值统计

**选择：** 明确区分"数值统计"和"语义分析"

**理由：**
- `len(posts)`、`sum(c["comments_count"])` 等不涉及文本理解
- 元信息的准确性有助于用户判断报告可信度
- 用户真正关心的是内容分析的质量

**边界定义：**
- ✅ 允许：数组长度、数值求和、百分比计算
- ❌ 禁止：正则匹配、词频统计、实体识别、情感分析脚本

## 风险 / 权衡

| 风险 | 缓解措施 |
|-----|---------|
| Agent 可能仍会调用脚本 | 使用 [核心要求] 标记，违反即任务失败 |
| 报告数字变得不准确 | 允许基本数值统计，元信息仍准确 |
| emoji 使用不一致 | 这是特性而非缺陷，让 AI 自由发挥 |
| 不同 Agent 行为差异大 | 约束是关于"禁止什么"，而非"必须怎么做" |

## 迁移计划

1. 修改 SKILL.md 文件
2. 无需数据迁移（仅指令变更）
3. 无回滚需求（如果效果不佳，可恢复原文档）
