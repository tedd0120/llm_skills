## 上下文

`login_xhs.py` 的 `_is_logged_in()` 方法当前采用反向验证：检查登录弹窗/二维码/登录按钮是否出现，都不出现则认为已登录。

问题：这种验证方式不可靠，无法确保搜索功能真正可用。

## 目标 / 非目标

**目标：**
- 实现正向验证：执行 test 搜索，验证搜索结果正常加载
- 确保登录验证能真正反映搜索功能可用性

**非目标：**
- 不修改登录流程的其他部分
- 不修改 fetch_xhs.py 的逻辑
- 不改变现有的选择器定义

## 决策

### 决策 1：验证标准

**选择**：无登录弹窗 + 搜索结果卡片存在

**理由**：
- 仅检查登录弹窗不可靠（Cookie 过期、网络问题可能导致无弹窗但功能不可用）
- 搜索 "test" 必定有结果，所以帖子卡片数量 > 0 是可靠的验证标准

**替代方案**：
- 使用热门关键词（如"美食"）：增加复杂性，"test" 已足够
- 检查 URL 是否重定向：小红书可能不重定向而是弹窗，不可靠

### 决策 2：修改位置

**选择**：仅修改 `_is_logged_in()` 方法

**理由**：
- 变更范围最小化
- 该方法被 `run()` 调用用于检查登录状态
- 不影响登录流程的其他部分

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 搜索结果加载慢导致超时 | 保持 10 秒超时与 fetch_xhs.py 一致 |
| 小红书改版导致选择器失效 | 选择器集中在 `xhs_selectors.py`，便于维护 |
