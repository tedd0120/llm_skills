## 上下文

**问题**：Agent 执行 xiaohongshu-scraper 时经常遗漏步骤（检查登录、格式化报告、发送报告）。当前的「核心要求」是静态文档声明，缺少运行时的强制检查机制。

**约束**：
- tasks.md 必须在 OUTPUT_DIR 创建后立即写入
- 验证脚本必须是外部脚本（非 Agent 自我检查）
- 不改变用户调用体验

**利益相关者**：xiaohongshu-scraper、xiaohongshu-summarize、xiaohongshu-formatter

## 目标 / 非目标

**目标：**
- 强制 Agent 在执行阶段按步骤打勾
- 阶段结束时验证所有任务完成
- 明确编排层与子 skill 的职责边界

**非目标：**
- 不改变 xiaohongshu-fetch 内部逻辑
- 不改变用户交互流程（澄清阶段保持不变）

## 决策

### 决策 1：tasks 内容粒度

**选择**：5 项粗粒度任务

```
- [ ] 检查登录状态
- [ ] 抓取数据
- [ ] 生成报告
- [ ] 格式化报告
- [ ] 发送报告
```

**理由**：
- 粗粒度任务对应编排层的 5 个原子操作
- 子 skill 内部的细节（如 fetch 的多轮循环）由子 skill 自己管理
- 避免任务列表过于冗长

**替代方案**：
- 细粒度拆分（10+ 项）：增加 Agent 维护负担，且容易与子 skill 内部逻辑耦合

### 决策 2：验证时机

**选择**：阶段结束时验证一次

**理由**：
- 中间状态允许未完成任务（正常流程）
- 只需在"发送报告"前确认所有步骤已完成
- 减少验证开销

**替代方案**：
- 每步完成后验证：验证 5 次，开销大且无必要

### 决策 3：职责边界

**选择**：scraper 负责编排，summarize 只负责生成

**理由**：
- 编排层统一调度所有子 skills
- summarize 可独立调用时仍能完成生成任务
- 避免 summarize 内部重复定义格式化和发送步骤

**影响**：
- 修改 `xiaohongshu-summarize/SKILL.md` 删除"输出要求"中的格式化和发送职责

### 决策 4：验证脚本位置

**选择**：放在 `xiaohongshu-scraper/scripts/verify_tasks.py`

**理由**：
- 该脚本专用于 scraper 的 tasks 验证
- 与 `login_xhs.py`、`fetch_xhs.py` 同目录，便于管理

## 风险 / 权衡

| 风险 | 缓解措施 |
|-----|---------|
| Agent 忘记打勾 | 核心要求明确规定"每完成一项必须更新 tasks.md" |
| tasks.md 格式错误 | verify_tasks.py 容错处理，统计 `[ ]` 和 `[x]` 数量 |
| 验证失败时 Agent 忽略 | 核心要求规定"验证失败必须报错中止" |
