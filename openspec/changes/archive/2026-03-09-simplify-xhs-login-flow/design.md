## 上下文

当前实现将登录相关行为拆成两个 CLI 使用方式：`--check-only` 仅验证 Cookie，默认模式在未登录时执行扫码登录。编排层通常需要先调用一次 `--check-only`，收到未登录结果后再调用一次默认模式，导致登录相关浏览器至少启动两次。与此同时，抓取组件已经被规范化为独立会话执行，因此本次更适合优化登录阶段的会话模型，而不是重构整个抓取生命周期。

## 目标 / 非目标

**目标：**
- 将“检查是否登录”和“必要时执行登录”收敛到单次浏览器会话中。
- 让编排层只关心登录是否已确保成功，不再承担探测后重试的分支判断。
- 保留 `--check-only` 作为独立诊断能力，但不再要求编排层在主流程中依赖它。

**非目标：**
- 不将登录阶段和抓取阶段合并到同一个浏览器实例。
- 不改变二维码登录、超时处理、Cookie 持久化等既有登录能力。
- 不调整 fetch、summarize、formatter 的接口和输出格式。

## 决策

### 决策 1：将默认登录命令定义为 ensure-login 语义

默认执行 `login_xhs.py` 时，组件应在同一个浏览器会话内：
1. 加载已有 Cookie；
2. 访问搜索结果页验证是否已登录；
3. 若已登录则直接返回成功；
4. 若未登录则继续展示二维码并等待扫码；
5. 成功后保存 Cookie 并返回成功。

选择该方案，是因为它在不破坏脚本边界的前提下去掉了一次冗余启动。相比继续维持“预检查 + 二次调用”，它简化了编排层状态处理；相比把登录和抓取合并成同一浏览器实例，它的改动范围更小、风险更低。

备选方案：
- 保持现状：职责清晰，但会持续保留重复会话开销。
- 登录与抓取完全合并：性能更好，但会把两个独立 CLI 脚本改造成共享运行时，超出本次范围。

### 决策 2：保留 `--check-only`，但降级为辅助手段

`--check-only` 仍然保留，用于诊断 Cookie 状态或手动排查问题，但编排层主流程不再依赖它。

选择该方案，是为了兼顾运维和调试便利性，避免把已有诊断入口完全移除。

### 决策 3：同步更新编排层规范，显式要求单次调用登录组件

`xiaohongshu-scraper` 的登录步骤将从“检查登录 → xiaohongshu-login --check-only”调整为“确保登录 → xiaohongshu-login”，由登录组件内部自行完成检查与必要登录。

选择该方案，是为了让能力边界与实际责任保持一致，避免规范仍然要求旧流程。

## 风险 / 权衡

- `[风险]` 已登录快速返回与扫码成功返回都可能映射为成功状态，编排层若依赖精细区分需同步梳理输出语义。 → 缓解措施：在规范中明确允许 `LOGIN_OK` 与 `LOGIN_SUCCESS` 都表示“可继续执行”。
- `[风险]` 旧的调用方如果仍强依赖 `--check-only` 再分支，短期内可能与新文档不一致。 → 缓解措施：同步更新 scraper 规范和技能文档，确保主路径统一。
- `[风险]` 默认模式内先检查再登录，会让登录脚本职责比之前稍重。 → 缓解措施：仍然限制在登录组件内部，不把抓取逻辑引入其中。

## Migration Plan

1. 更新 `xhs-login` 增量规范，定义默认模式的单次会话行为，并保留 `--check-only` 作为可选诊断模式。
2. 更新 `xiaohongshu-scraper` 增量规范，改写编排层的登录步骤说明。
3. 后续实现时同步修改 `login_xhs.py` 和 `xiaohongshu-scraper/SKILL.md`，并验证未登录、已登录、超时三条主路径。
