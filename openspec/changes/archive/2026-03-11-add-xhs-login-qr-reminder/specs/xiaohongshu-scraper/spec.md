## 新增需求

### 需求: 登录步骤必须使用后台轮询模式
scraper 在执行阶段的“确保登录”步骤必须以后台运行 + 轮询的方式调用 `xiaohongshu-login`，以便在扫码等待期间向用户发送提示。

#### 场景: 后台启动登录任务
- **当** scraper 进入执行阶段的登录步骤
- **那么** 必须以后台模式启动登录脚本
- **并且** 禁止使用阻塞模式等待脚本结束后再处理输出

#### 场景: 轮询间隔可配置
- **当** scraper 轮询登录任务输出
- **那么** 必须支持 `LOGIN_POLL_INTERVAL_SEC` 配置项
- **并且** 默认值为 2 秒
- **并且** 建议配置范围为 1-10 秒
- **并且** 配置值非法时必须回退到默认值 2 秒

### 需求: NEED_LOGIN 事件必须触发扫码提醒
scraper 在轮询过程中检测到 `NEED_LOGIN:<path>` 时，必须立即向用户发送扫码提示，并展示二维码文件路径。

#### 场景: 首次检测到 NEED_LOGIN
- **当** 轮询新增输出中出现 `NEED_LOGIN:<path>`
- **那么** Agent 必须立即提示用户扫码登录
- **并且** 提示内容必须包含二维码图片绝对路径 `<path>`

#### 场景: 扫码后收敛
- **当** 轮询新增输出出现 `LOGIN_OK` 或 `LOGIN_SUCCESS`
- **那么** scraper 必须将“确保登录”步骤标记为完成
- **并且** 继续执行抓取步骤

#### 场景: 登录失败收敛
- **当** 轮询新增输出出现 `LOGIN_TIMEOUT` 或 `LOGIN_FAILED`
- **那么** scraper 必须报告错误并中止执行阶段
- **并且** 禁止继续抓取步骤

## 修改需求

### 需求: 编排层职责明确定义
xiaohongshu-scraper 作为编排层，必须明确定义其职责为：检查登录、调用 fetch 抓取数据、调用 summarize 生成报告、调用 formatter 格式化报告、发送报告。

#### 场景: 编排层职责清晰
- **当** Agent 执行 xiaohongshu-scraper
- **那么** Agent 必须按顺序执行：检查登录 → 抓取数据 → 生成报告 → 格式化报告 → 发送报告
- **并且** 登录步骤若进入扫码等待，必须先向用户发送扫码提示（含二维码绝对路径）再继续等待结果
- **并且** 每个步骤完成后必须更新 tasks.md
