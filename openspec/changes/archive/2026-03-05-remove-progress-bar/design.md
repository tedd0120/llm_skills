## 上下文

当前 `xiaohongshu-scraper` 技能在 Agent 执行步骤 2 中包含一个复杂的进度条渲染系统：
- Agent 通过 `TaskOutput` 轮询后台任务输出
- 解析脚本输出的 `[i/total]` 格式
- 渲染可视化进度条（`████░░░░░░░░░░░ 17%`）
- 实现去重逻辑避免重复显示相同进度

脚本端 (`fetch_xhs.py`) 已经输出详细的文本进度信息：
```
[*] 搜索关键词: xxx (配额 N, 总剩余 M)
  找到 X 篇（去重后），抓前 Y 篇
  [i/total] URL...
[✓] 共 N 篇 → output_file
```

## 目标 / 非目标

**目标：**
- 简化 Agent 执行逻辑，移除不必要的可视化进度条渲染
- 保持脚本端输出格式不变（兼容性）
- 保持后台任务启动和完成处理机制

**非目标：**
- 不修改脚本端行为
- 不改变最终报告生成流程

## 决策

### 1. 移除范围界定

**决策**：仅移除 Agent 端的进度条渲染逻辑，保留脚本端输出。

**理由**：
- 脚本端输出格式 (`[i/total]`) 对其他潜在调用者（如直接使用 CLI）仍然有价值
- 修改脚本端可能影响其他依赖该输出的场景
- Agent 可以通过 `block=true` 的 `TaskOutput` 等待完成，无需中间进度

**替代方案考虑**：
- 同时修改脚本端输出为静默模式 → 被拒绝，会降低 CLI 使用体验

### 2. 替代进度反馈

**决策**：删除后，Agent 在任务完成时向用户展示简短完成消息。

**理由**：
- 用户仍需知道任务已完成
- 最终报告 (`_index.md`) 生成后已包含详细信息

## 风险 / 权衡

| 风险 | 缓解措施 |
|-----|---------|
| 用户失去进度反馈，无法感知任务是否在运行 | 任务完成时会输出完成消息；脚本端输出仍在后台任务日志中可见 |
| 长时间运行任务时用户焦虑 | 此为预期权衡；简化逻辑优先 |

## 具体实现

需要从 `SKILL.md` 中删除的内容（步骤 2.2）：
- 轮询循环伪代码
- `renderProgressBar` 函数
- 进度条展示效果示例
- `extractLatestProgress` / `extractCurrentKeyword` 解析函数
- 静默轮询规则
- `[核心要求] 必须在轮询循环中主动向用户展示进度`

简化后的步骤 2 结构：
- 2.1 启动后台抓取任务（保持不变）
- 2.2 等待完成并处理结果（使用 `TaskOutput` with `block=true`）
