## 上下文

### 当前状态
- `fetch_xhs.py` 使用 `print()` 输出进度信息，但由于 Python 的 stdout 缓冲机制，输出可能延迟显示
- Agent 通过 Bash 工具以阻塞模式调用 Python 脚本，必须等待脚本完全结束才能获取输出
- 用户在抓取过程中看不到任何反馈，体验不佳

### 约束
- 不能修改 Bash 工具本身的行为
- Python 脚本需要保持跨平台兼容性（Windows/Linux）
- 进度反馈不能显著增加执行时间开销

### 利益相关者
- **用户**：需要知道抓取进度，避免"盲等"
- **Agent**：需要在后台运行时能够解析进度信息

## 目标 / 非目标

**目标：**
- 用户能够在抓取过程中看到实时进度（当前关键词、已完成篇数/总篇数）
- Agent 能够主动向用户同步抓取进度
- Python 脚本输出不被缓冲

**非目标：**
- 实现精确的进度百分比（由于去重和动态配额分配，精确百分比不可行）
- 提供剩余时间估算
- 修改 Bash 工具的行为

## 决策

### 决策 1: 使用后台运行 + 轮询模式

**选择理由：**
- Bash 工具支持 `run_in_background=true`，返回 task_id
- `TaskOutput(task_id, block=false)` 可以持续读取部分输出
- Agent 可以在循环中轮询进度并向用户展示

**替代方案考虑：**
| 方案 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| 后台运行 + 轮询 | 实时反馈，Agent 可主动报告 | 需要轮询逻辑 | ✅ 采用 |
| 分阶段调用 | 每个关键词独立调用 | 多次浏览器启动开销大 | ❌ 拒绝 |
| 进度文件写入 | Agent 可独立读取 | 有文件 I/O 延迟 | ❌ 拒绝 |
| 保持阻塞模式 | 无需修改代码 | 用户体验差 | ❌ 拒绝 |

### 决策 2: Python 输出添加 flush=True

**选择理由：**
- 强制刷新 stdout 缓冲区，确保输出立即可见
- 跨平台兼容（Windows/Linux）
- 性能开销可忽略

**代码变更点：**
```python
# 所有进度输出语句
print(f"[*] 搜索关键词: {kw}...", flush=True)
print(f"  [{i+1}/{total}] {url}...", flush=True)
```

### 决策 3: 进度输出格式规范

**格式设计：**
```
[*] 搜索关键词: xxx (配额 N, 总剩余 M)
  找到 X 篇（去重后），抓前 Y 篇
  [i/total] URL...
[✓] 共 N 篇 → output_file
```

**解析规则：**
- `[*] 搜索关键词:` → 关键词级别进度
- `[n/m]` → 单篇进度
- `[✓] 共` → 完成标志

### 决策 4: 进度去重逻辑（新增）

**问题背景：**
轮询间隔（5秒）可能短于 Python 脚本的实际进度变化（防反爬延时 3-8秒），导致相同内容重复输出 40+ 次，用户体验不佳。

**去重策略：**
```
轮询循环伪代码：
last_output_hash = ""
last_shown_progress = ""

while True:
    output = TaskOutput(task_id, block=false)

    # 计算输出内容的哈希或比较最后一行
    current_progress = extractLastProgressLine(output)

    # 只在进度变化时显示
    if current_progress != last_shown_progress:
        renderProgressBar(output)
        last_shown_progress = current_progress

    if isCompleted(output): break
    sleep(5)
```

**进度条格式：**
```
[2/6] ████████░░░░░░░░ 33% 北京咖啡厅推荐
```

**去重效果：**
- 无去重：40+ 条输出（大量重复）
- 有去重：~10 条输出（只在变化时更新）

**[重要] 静默轮询规则：**
在轮询过程中，当进度未变化时，**禁止输出任何消息**。这意味着：
- 不输出"等待中..."、"轮询中..."、"正在检查..."等任何状态消息
- 每次轮询时只检查进度是否变化
- 仅当 `current_progress != last_shown_progress` 时才输出
- 否则完全静默，直到下次轮询

**预期输出（6篇帖子）：**
```
[1/6] ████░░░░░░░░░░░ 17% 北京咖啡厅推荐
[2/6] ██████░░░░░░░░░ 33% 北京咖啡厅推荐
[3/6] ████████░░░░░░ 50% 北京咖啡厅探店
[4/6] ██████████░░░░ 67% 北京咖啡厅探店
[5/6] ████████████░░ 83% 北京咖啡厅避坑
[6/6] ██████████████ 100% 完成！
```
**只有 6 条输出，中间无任何等待消息。**

### 决策 5: 进度条展示格式

**选择理由：**
- 用户无法看到真正的"动态"进度条（Agent 消息独立，无法覆盖）
- 使用 ASCII 字符 `█` 和 `░` 构建可视化进度
- 单行紧凑格式，减少视觉干扰

**格式：**
```
[当前/总数] 进度条 百分比 当前关键词
```

**示例：**
```
[1/6] ████░░░░░░░░░░░ 17% 北京咖啡厅推荐
[2/6] ██████░░░░░░░░░ 33% 北京咖啡厅推荐
[3/6] ████████░░░░░░ 50% 北京咖啡厅探店
...
[6/6] ██████████████ 100% 完成！
```

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 轮询间隔过短增加开销 | 设置合理间隔（5秒） |
| Python 缓冲问题仍然存在 | 使用 `flush=True` + `-u` 参数 |
| 进度解析可能失败 | 使用容错的正则匹配，失败则显示原始输出 |
| 后台任务可能失败退出 | 检查退出码，非零时报告错误 |
| 进度重复显示 | 添加去重逻辑，只在变化时更新显示 |
| 进度条与内容不匹配 | 从多个输出行综合判断当前状态 |
| Agent 输出等待消息 | **[核心要求]** 在 skill.md 中明确禁止轮询时的任何输出，只在进度变化时输出 |
