## 上下文

现有登录脚本在未登录时会输出 `NEED_LOGIN:<qr_path>` 并继续阻塞等待扫码。该输出本身足以触发提醒，但由于编排层以阻塞方式调用脚本，Agent 无法在等待期间向用户发送消息，导致“二维码已生成但用户无感知”。

历史上抓取阶段曾引入复杂进度轮询并被移除。本次仅针对登录阶段引入最小事件轮询，不恢复可视化进度条，也不增加抓取阶段复杂性。

## 目标 / 非目标

**目标：**
- 在未登录时让 Agent 尽快提醒用户扫码，并明确二维码路径。
- 轮询间隔可配置，便于在不同设备和负载下调整。
- 保持登录状态机语义稳定，避免影响既有调用方。

**非目标：**
- 不改造抓取阶段为实时进度展示。
- 不新增登录状态码。
- 不改变 `login_xhs.py` 的扫码、超时和 Cookie 持久化主流程。

## 决策

### 决策 1：登录步骤改为后台执行 + 事件轮询

编排层在执行“确保登录”时，必须以后台任务方式启动 `login_xhs.py`，并按固定间隔轮询新增输出。

轮询关注的事件只有 5 类：
- `NEED_LOGIN:<abs_path>`
- `LOGIN_OK`
- `LOGIN_SUCCESS`
- `LOGIN_TIMEOUT`
- `LOGIN_FAILED`

选择原因：可在不改状态码的前提下立即感知扫码需求，改动范围最小。

### 决策 2：轮询间隔配置化

新增配置项：
- `LOGIN_POLL_INTERVAL_SEC`（默认 2 秒，建议范围 1-10 秒）

行为规则：
- 未配置时使用默认值 2 秒。
- 配置非法值时回退默认值并记录说明。

选择原因：2 秒可以较快发现 `NEED_LOGIN`，同时避免过高轮询频率带来的无谓开销。

### 决策 3：扫码提示必须展示二维码绝对路径

首次观测到 `NEED_LOGIN:<abs_path>` 时，Agent 必须立即发送提示，包含：
- 明确动作：请扫码登录
- 二维码绝对路径：`<abs_path>`

为避免重复刷屏，允许实现提醒冷却策略，但不作为本次强制要求。

选择原因：路径是用户可直接定位二维码文件的最关键信息，必须外显。

### 决策 4：xhs-login 输出时序要求

当进入扫码路径时，`xhs-login` 必须先输出 `NEED_LOGIN:<abs_path>`，再进入等待扫码阶段。该要求确保编排层在轮询中能尽早获取提醒事件。

## 状态机草图

```text
          ┌────────────────────────┐
          │ 启动 login 后台任务    │
          └──────────┬─────────────┘
                     ▼
          ┌────────────────────────┐
          │ 轮询新增输出(每 N 秒)  │
          └──────────┬─────────────┘
                     │
        ┌────────────┼──────────────┬──────────────┬──────────────┐
        ▼            ▼              ▼              ▼              ▼
  NEED_LOGIN     LOGIN_OK      LOGIN_SUCCESS   LOGIN_TIMEOUT   LOGIN_FAILED
        │            │              │              │              │
        ▼            ▼              ▼              ▼              ▼
 提示扫码+路径     登录完成        登录完成        报错退出        报错退出
        │
        └───────────────回到轮询，等待最终成功/失败─────────────────────┘
```

## 风险 / 权衡

| 风险 | 缓解措施 |
|-----|---------|
| 轮询间隔过长，提醒变慢 | 默认 2 秒并支持配置 |
| 轮询间隔过短，产生不必要开销 | 限制推荐范围 1-10 秒 |
| `NEED_LOGIN` 文本格式被破坏导致无法解析 | 规范中固定输出格式，解析仅按前缀匹配 |
| 终端路径显示与用户预期不一致 | 明确要求输出绝对路径 |
