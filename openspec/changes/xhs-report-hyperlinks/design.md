## 上下文

### 当前状态
`xiaohongshu-scraper` 技能生成的 `_index.md` 报告中，原帖引用采用纯文本格式（如 `*《帖子标题》*`）。用户需要手动复制标题去小红书搜索来验证或深度阅读原帖。

### 约束条件
- 变更必须向后兼容，默认行为保持不变
- 不能引入外部依赖（如短链接服务）
- 需要在澄清阶段增加用户选择交互
- 现有的 fetch → summarize → formatter 流程需保持

### 利益相关者
- 用户：需要便捷地访问原帖进行验证
- Agent 执行者：需要清晰的指令处理新的配置选项

## 目标 / 非目标

**目标：**
- 为报告中的原帖引用添加可选的超链接功能
- 用户可在澄清阶段选择是否启用超链接
- 使用关注点分离的 ID 映射架构，便于未来扩展
- 完全向后兼容，默认关闭

**非目标：**
- 不改变报告的核心内容分析逻辑
- 不引入短链接服务或外部 API
- 不修改现有的数据抓取行为

## 决策

### 决策 1: 使用完整 URL 而非短链接

**选择：** 直接在小红书原始 URL 作为超链接目标

**理由：**
- 无需外部依赖，不增加成本
- Markdown 超链接中 URL 长度不影响显示效果
- 避免第三方服务不稳定或下线的风险

**替代方案：**
- 短链接服务（TinyURL、新浪短链等）：增加依赖和复杂度，被否决

### 决策 2: ID 映射分离架构

**选择：** 方案 B —— 使用独立的 `id_url_map.json` 文件存储 ID 到 URL 的映射

**理由：**
- **关注点分离**：summarize 阶段只关心内容分析，不关心 URL 格式
- **灵活性**：将来 URL 格式变更（如短链接）只需修改 formatter
- **易于测试**：URL 替换逻辑独立，可单独验证

**架构：**
```
fetch 阶段: 生成 raw.json (含 post_id) + id_url_map.json
summarize 阶段: 生成含 ID 占位符的报告 [*《标题》*](id:xxxxx)
formatter 阶段: 读取 id_url_map.json，替换 ID 为 URL
```

**替代方案：**
- 方案 A（URL 直接存储在 raw.json）：简单但缺乏灵活性，被否决

### 决策 3: 用户交互设计

**选择：** 每次询问用户，默认关闭

**理由：**
- 保守策略，不影响现有用户习惯
- 不同报告场景可能有不同需求

**交互流程：**
```
步骤 1: 模式选择（A 固定 / B 发散）
步骤 2: 关键词/参数确认
步骤 3: 超链接格式选择 ← 新增
  A: 纯文本（默认）
  B: 超链接
```

### 决策 4: 配置传递方式

**选择：** 通过环境变量和文件存在性传递配置

**理由：**
- 简单可靠，无需额外配置文件
- `id_url_map.json` 存在即表示超链接功能启用

**传递机制：**
```python
# fetch 阶段
if hyperlinks_enabled:
    生成 id_url_map.json

# summarize 阶段
根据 hyperlinks 配置决定引用格式

# formatter 阶段
if os.path.exists("id_url_map.json"):
    执行 URL 替换
```

### 决策 5: 帖子 ID 提取方式

**选择：** 从小红书 URL 路径中提取

**理由：**
- 小红书 URL 格式稳定：`https://www.xiaohongshu.com/explore/{note_id}`
- note_id 具有唯一性，可作为映射键

**提取逻辑：**
```python
url = "https://www.xiaohongshu.com/explore/6543210abcd"
post_id = url.rstrip("/").split("/")[-1]  # "6543210abcd"
```

### 决策 6: 占位符格式

**选择：** `[*《帖子标题》*](id:{post_id})`

**理由：**
- 格式清晰，易于正则匹配替换
- `id:` 前缀避免与实际 URL 冲突
- 保持原有的书名号样式

**替代方案：**
- `#post_id`、`@post_id`：格式不够明确

## 风险 / 权衡

### 风险 1: 旧数据兼容性
**风险：** 旧的 raw.json 没有 post_id 字段，可能导致错误

**缓解措施：**
- summarize 阶段检测 post_id 是否存在，不存在时降级为纯文本格式
- formatter 阶段检测 id_url_map.json 是否存在，不存在时跳过替换

### 风险 2: URL 格式变化
**风险：** 小红书 URL 格式变化导致 ID 提取失败

**缓解措施：**
- 使用 URL 路径提取而非正则匹配，容错性更强
- 提取失败时记录警告但不中断流程

### 风险 3: 配置传递复杂度
**风险：** hyperlinks 配置在多个阶段间传递，可能出现不一致

**缓解措施：**
- 使用文件存在性作为明确信号
- 在 SKILL.md 中清晰定义各阶段行为

### 权衡: 额外文件 vs 灵活性
**权衡：** 引入 id_url_map.json 增加了一个文件

**收益：**
- 架构更清晰，职责分离
- 未来 URL 格式变更成本更低
- 便于调试和测试

**结论：** 收益大于成本，接受此权衡
